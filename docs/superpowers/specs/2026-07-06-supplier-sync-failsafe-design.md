# Авто-снятие товаров поставщика с наличия при падении синхронизации

Дата: 2026-07-06
Статус: одобрено

## Проблема

Крон «Клипса - сохранить в БД актуальную информацию о товарах поставщиков»
(`saveToMongoUnifiedProducts`) при падении API поставщика (пример: ЕРЦ,
connect.erc.ua отдаёт 500 с 05.07.2026, данные не обновлялись с 18.06.2026)
оставляет в `products_unified` устаревшее наличие и цены. Товары продолжают
продаваться на сайте по неактуальным данным.

## Решение

Если синхронизация конкретного поставщика завершилась ошибкой (на любом
этапе — фетч или запись в БД), все его товары в `products_unified`
переводятся в `availability: false`. Триггер — первая же ошибка (решение
пользователя от 06.07.2026). Поставщики независимы: падение одного не влияет
на других.

Самовосстановление: следующая успешная синхронизация возвращает наличие через
существующую логику `isProductChanged` (сравнение `availability` БД ↔ API).

## Изменения

1. **`IDatabase` + `DatabaseMongo`**: новый метод
   `makeSupplierProductsUnavailable(supplierName: string): Promise<number>` —
   `updateMany({supplier_name, availability: true}, {$set: {availability: false}})`
   по `products_unified`, возвращает количество обновлённых документов.
   Обработка ошибок в стиле остальных методов (`log.all` + throw).

2. **`SuppliersApiAbstraction.saveProductsToDb()`**: цепочка `save()` каждого
   поставщика получает catch: при ошибке вызывается
   `makeSupplierProductsUnavailable(supplierName)`, затем ошибка
   пробрасывается с дополнением:
   - успех зануления: `Все товары поставщика (N шт.) переведены в "Нет в наличии"`;
   - ошибка зануления: `НЕ удалось перевести товары поставщика в "Нет в наличии": <причина>`.

   Текст попадает в `reason` лога крона (`Logs.cron_jobs`).

3. **Тесты**: встроенный `node:test` (без новых зависимостей), скрипт
   `npm test` через ts-node. Юнит-тесты `SuppliersApiAbstraction` с фейковыми
   поставщиком/БД:
   - упавший поставщик → зануление вызвано только для него, ошибка проброшена с припиской;
   - успешный поставщик → зануление не вызывается, товары сохранены;
   - ошибка самого зануления → исходная ошибка дополнена, не проглочена.

   `DatabaseMongo.makeSupplierProductsUnavailable` (тривиальный updateMany)
   проверяется вручную; сам запрос проверен в бою 06.07.2026 (разовое снятие
   27 654 товаров ЕРЦ с наличия).

## Вне скоупа

- Случай «весь сервис clipsa лежит» — никто не занулит наличие. Покрывается
  отдельной задачей: `last_success_at` по поставщику + проверка свежести в
  content API (вариант C из брейншторма).
- Замена `throw <строка>` на `throw new Error(...)` в клиентах ЕРЦ
  (причина «Unknown error» в логах) — отдельная мелкая задача.
